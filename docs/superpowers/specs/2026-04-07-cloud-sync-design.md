## Overview

Server-side support for TriggerTime multi-device cloud sync. The server stores synced shooting session data in relational tables, provides push/pull API endpoints for mobile devices, and displays sessions in a read-only web dashboard.

## Sync API Endpoints (Mobile)

All under `/api/v1/sync/`, authenticated via HMAC API key middleware. Server must verify:
1. Device exists and is linked to a user (`user_id` is not null on `devices` table)
2. User has an active PRO subscription

### `POST /api/v1/sync/push`

**Request:**
```json
{
  "device_uuid": "abc-123",
  "records": {
    "disciplines": [
      {
        "uuid": "...",
        "name": "Custom Discipline",
        "weapon_type_id": 1,
        "scoring_type_id": 1,
        "modified_at": "2026-04-07T10:00:00Z",
        "deleted": false,
        "...other fields"
      }
    ],
    "phases": [...],
    "sessions": [...],
    "series": [...],
    "shots": [...],
    "strings": [...]
  }
}
```

**Server logic per record:**
1. Look up existing record by UUID
2. If not found â†’ insert, set `user_id` from device's linked user
3. If found and request `modified_at` > server `modified_at` â†’ update
4. If found and request `modified_at` <= server `modified_at` â†’ reject (server newer)
5. If `deleted: true` â†’ set `deleted_at` on server

**Processing order:** disciplines â†’ phases â†’ sessions â†’ series â†’ shots/strings (respect FK dependencies)

**Response:**
```json
{
  "accepted": ["uuid1", "uuid2"],
  "rejected": [
    { "uuid": "uuid3", "reason": "server_newer", "server_modified_at": "2026-04-07T11:00:00Z" }
  ],
  "last_sync_at": "2026-04-07T12:00:00Z"
}
```

### `GET /api/v1/sync/pull`

**Query params:**
- `device_uuid` (required) â€” requesting device
- `since` (required) â€” ISO 8601 timestamp, pull records modified after this

**Server logic:**
1. Verify device ownership + active subscription
2. Query all `sync_*` tables for this user where `modified_at > since`
3. Include soft-deleted records (`deleted_at IS NOT NULL`) so app can delete locally
4. Support pagination: return max 500 records per table, include `has_more: true` if more exist

**Response:**
```json
{
  "records": {
    "disciplines": [...],
    "phases": [...],
    "sessions": [...],
    "series": [...],
    "shots": [...],
    "strings": [...]
  },
  "has_more": false,
  "sync_timestamp": "2026-04-07T12:00:00Z"
}
```

### `GET /api/v1/sync/status`

**Query params:**
- `device_uuid` (required)

**Response:**
```json
{
  "has_changes": true,
  "last_sync_at": "2026-04-07T12:00:00Z"
}
```

Lightweight check â€” counts records where `modified_at > device's last pull timestamp`. Used by app to show sync indicator without downloading data.

## Web Dashboard â€” Read-Only Sessions

### New Web API Endpoints (JWT auth)

**`GET /api/v1/web/sessions`**
- Query params: `sort` (date asc/desc), `discipline` (filter), `type` (filter), `page`, `limit`
- Queries `sync_sessions` where `user_id = authenticated user` and `deleted_at IS NULL`
- Joins discipline name (denormalized field)
- Returns paginated list

**`GET /api/v1/web/sessions/:uuid`**
- Returns full session detail with nested series/shots/strings
- Verifies `user_id` matches authenticated user
- Includes phase information from `sync_phases` if available

### New Vue SPA Pages

**`/dashboard/sessions`** â€” Sessions list
- Sortable table: date, discipline, type, location, total score, x-count
- Default sort: newest first
- Filters: discipline dropdown, session type dropdown

## Error Handling

- **Push with invalid device/subscription:** Return 403 with `{ error: "sync_not_authorized", message: "..." }`
- **Push with FK violation (e.g., session references non-existent discipline UUID):** Return 422 with details, skip that record, continue processing others
- **Pull with no changes:** Return empty `records` object, `has_more: false`
- **Rate limiting:** Consider throttling sync endpoints to prevent abuse (e.g., max 60 pushes/minute per device)
