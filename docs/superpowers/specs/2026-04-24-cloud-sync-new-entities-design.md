## Overview

Extend the existing cloud sync system to support 5 new entity types: **Weapon**, **Ammo**, **Competition**, **CompetitionReminder**, and **AmmoTransaction**. Also add 3 new UUID FK columns to Session (`weapon_uuid`, `ammo_uuid`, `competition_uuid`) for cross-device FK resolution.

The new entities follow the same patterns as the original sync: UUID primary keys, `modified_at`/`deleted_at` soft-delete fields, last-modified-wins conflict resolution, and push/pull through the existing `/api/v1/sync/push` and `/api/v1/sync/pull` endpoints.

### Ownership Filter Extension

The existing `applyOwnershipFilter()` method uses a `switch` on `$config['ownership']`. Two new ownership strategies are needed:

- **`via_competition`**: Join `sync_competition_reminders` â†’ `sync_competitions` on `competition_uuid = id` and filter by `sync_competitions.user_id`
- **`via_ammo`**: Join `sync_ammo_transactions` â†’ `sync_ammo` on `ammo_uuid = id` and filter by `sync_ammo.user_id`

These follow the exact pattern of existing `via_discipline`, `via_session`, and `via_series` strategies.

### Provider Updates

Each provider (WeaponProvider, AmmoProvider, CompetitionProvider) must use `syncAwareInsert`, `syncAwareUpdate`, and `syncAwareDelete` instead of raw DB calls. These methods auto-generate UUIDs, set `modified_at`, and set `sync_status = 1` (pending push).

After each mutation, call `SyncService().triggerSync()` to queue a sync attempt.

### SyncService Updates (Flutter)

The Flutter `SyncService` must be updated to:
1. **Push**: Collect pending records from the 5 new tables (where `sync_status = 1` or `sync_status = 2`), build sync maps with UUID FK resolution, and include them in the push payload
2. **Pull**: Process incoming records for the 5 new types â€” insert/update local records, resolve UUID-to-local-ID mappings for FK fields
3. **UUID FK resolution on push**: When building the Session sync map, look up the `weapon_uuid`, `ammo_uuid`, and `competition_uuid` from the local integer FK IDs
4. **UUID FK resolution on pull**: When receiving a Session with `weapon_uuid`/`ammo_uuid`/`competition_uuid`, resolve to local integer IDs by looking up the UUID in the local tables

## shot_count Sync Strategy

`shot_count` on Weapon is both user-editable and auto-incremented when a session finishes. The sync strategy is:

- **Sync as-is (last-modified-wins):** The `shot_count` value is synced like any other field. The device with the most recent `modified_at` wins.
- **User can override at any moment:** Users can manually edit `shot_count` on the weapon form screen, which updates `modified_at` and triggers sync.
- **Auto-increment:** When a session finishes, the provider increments `shot_count` and updates `modified_at`, which triggers sync.

This means if two devices both finish sessions around the same time, the last one to sync wins and the other device's increment is lost. This is acceptable because:
1. Single-device usage is the common case
2. The user can always correct `shot_count` manually
3. Adding CRDT or increment-merge logic would add significant complexity for a minor edge case

## Processing Order Rationale

```
disciplines â†’ phases â†’ weapons â†’ ammo â†’ competitions â†’ competition_reminders â†’ sessions â†’ series â†’ shots â†’ strings â†’ ammo_transactions
```

1. **Disciplines/Phases first** â€” sessions reference disciplines via UUID
2. **Weapons, Ammo, Competitions** â€” sessions reference these via UUID. No dependencies between them, but they must come before sessions
3. **Competition Reminders** â€” depends on competitions
4. **Sessions** â€” depends on disciplines, weapons, ammo, competitions
5. **Series/Shots/Strings** â€” depends on sessions
6. **Ammo Transactions last** â€” depends on ammo, sessions, and weapons (all must exist)

## Error Handling

Same as existing sync spec:
- Push with FK violation (e.g., ammo_transaction references non-existent ammo UUID): Return 422, skip that record, continue processing
- Invalid ownership (user doesn't own the parent record): Return 403 for that record
- Missing referenced UUIDs on pull (e.g., Session has `weapon_uuid` but weapon was deleted): Set FK to null, don't fail
